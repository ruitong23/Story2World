目标：
在现有 Story2World / 小说世界模拟器程序外面，新增一个独立的 API Server 层，供前端网站调用。

这个 API 层不要重构现有 Step 1-16 主流程，不要改抽取逻辑，不要新建平行测试流程。它只负责：
1. 接收用户上传的 txt 小说文件
2. 创建 project_id
3. 调用现有 pipeline 处理小说
4. 记录处理状态和进度
5. 处理完成后返回可供前端展示的数据
6. 接收用户对话输入
7. 调用已有 agent_profiles / world_db / character_state_db / structured_world_graph / hybrid search 生成回复
8. 返回人物关系表、角色列表、世界总览、对话结果等 UI 需要的数据

一、技术选择

使用 FastAPI 作为后端 API。
使用本地文件系统保存每个 project 的数据。
暂时不需要数据库，不需要用户系统，不需要 Celery，不需要复杂部署。
任务执行可以先用 BackgroundTasks / thread / process 实现。
本地 LLM 继续使用 OpenAI Compatible API，例如 LM Studio 的 http://localhost:1234/v1。

三、API 需求

1. 创建项目并上传小说

POST /projects

功能：
- 接收 multipart/form-data 文件
- 只允许 .txt
- 文件大小限制 6MB
- 自动生成 project_id
- 保存为 projects/{project_id}/raw.txt
- 初始化 status.json
- 启动后台 pipeline
- 立即返回 project_id，不等待处理完成

返回示例：

{
  "project_id": "project_xxx",
  "status": "queued"
}

2. 查询处理状态

GET /projects/{project_id}/status

功能：
- 读取 projects/{project_id}/status.json
- 返回当前状态、当前步骤、进度、错误信息

状态类型：
queued
processing
ready
failed

返回示例：

{
  "project_id": "project_xxx",
  "status": "processing",
  "current_step": "Step 9 Graph Triple Extraction",
  "progress": 0.42,
  "message": "正在抽取世界图谱"
}

完成示例：

{
  "project_id": "project_xxx",
  "status": "ready",
  "progress": 1.0,
  "message": "小说世界已准备完成"
}

3. 获取 Dashboard 总览

GET /projects/{project_id}/dashboard

功能：
- 只在 ready 后可用
- 读取 world_db.json、character_state_db.json、agent_profiles.json、structured_world_graph.json
- 返回前端首页总览需要的数据

需要包括：
- 项目状态
- 角色数量
- 可选 agent 数量
- 地点数量
- 事件数量
- 关系数量
- 主要角色列表
- 当前世界进度
- 可用功能

返回示例：

{
  "project_id": "project_xxx",
  "status": "ready",
  "characters_count": 42,
  "agents_count": 12,
  "locations_count": 18,
  "events_count": 120,
  "relationships_count": 260,
  "main_characters": [],
  "available_features": ["chat", "relationships", "characters", "world"]
}

4. 获取角色列表

GET /projects/{project_id}/characters

功能：
- 读取 agent_profiles.json 和 character_state_db.json
- 返回可对话角色和基础角色信息

返回字段建议：
character_id
name
aliases
titles
short_description
available_as_agent
relationship_count
ability_count
item_count

5. 获取人物关系表

GET /projects/{project_id}/relationships

功能：
- 从 structured_world_graph.json / character_state_db.json 中整理人物关系
- 返回前端可直接渲染的 table rows

返回字段：
source_character
relationship_type
target_character
description
confidence
source_event
source_text
canonical_source_id
canonical_target_id

注意：
不要在这里重新抽取关系。
只能使用已有 structured graph / character_state_db 的结果做整理。

6. 获取世界信息

GET /projects/{project_id}/world

功能：
- 读取 world_db.json
- 返回地点、组织、事件、规则、能力系统、物品系统、时间线等

7. 创建或继续对话

POST /projects/{project_id}/chat

输入：

{
  "session_id": "session_001",
  "character_id": "char_xxx",
  "message": "你现在认识谁？"
}

功能：
- 检查 project 是否 ready
- 读取 agent_profiles.json
- 读取 character_state_db.json
- 读取 world_db.json
- 读取 structured_world_graph.json
- 读取 session 历史
- 使用 hybrid search 检索相关原文 / 世界信息
- 构建 runtime prompt
- 调用本地 LLM
- 保存本轮对话到 sessions/{session_id}.json
- 返回角色回复

返回示例：

{
  "session_id": "session_001",
  "character_id": "char_xxx",
  "reply": "...",
  "used_sources": [],
  "world_constraints": [],
  "related_relationships": [],
  "state_delta": {}
}

四、Pipeline 封装要求

需要把现有 notebook / 脚本里的主流程封装成一个函数：

run_full_pipeline(project_dir: str, progress_callback=None)

输入：
projects/{project_id}/

读取：
raw.txt

输出：
project_data.json
novel_ontology.json
raw_graph_triples.json
mention_alias_index.json
canonical_entities.json
normalized_graph_triples.json
structured_world_graph.json
character_state_db.json
world_db.json
agent_profiles.json

要求：
- 不要重写 Step 1-16
- 只做函数封装
- 每完成一个阶段，通过 progress_callback 更新 status.json
- 如果中间失败，写入 failed 状态和错误信息
- 支持重复运行时检查已有文件，但第一版可以简单覆盖 project 目录内的中间结果

五、状态文件格式

projects/{project_id}/status.json：

{
  "project_id": "project_xxx",
  "status": "processing",
  "current_step": "Step 9 Graph Triple Extraction",
  "progress": 0.42,
  "message": "正在抽取世界图谱",
  "error": null,
  "created_at": "...",
  "updated_at": "..."
}

六、前端对接方式

前端只需要知道 API_BASE。

本地开发：

const API_BASE = "http://localhost:8000"

前端流程：

1. 用户上传 txt
2. 调 POST /projects
3. 拿到 project_id
4. 轮询 GET /projects/{project_id}/status
5. status = ready 后进入 dashboard
6. dashboard 调：
   GET /dashboard
   GET /characters
   GET /relationships
   GET /world
7. 用户聊天时调：
   POST /chat

七、限制

第一版限制：
- 单文件最大 6MB
- 只支持 txt
- 不做登录
- 不做多人并发优化
- 不做数据库
- 不做收费
- 不做复杂权限
- 不做 websocket
- 不做流式输出
- 不做多服务器部署

八、不要做的事

不要改现有 Step 1-16 的抽取主线。
不要把 API 逻辑写进抽取步骤里。
不要让前端直接读 json 文件。
不要让前端直接访问本地 LLM。
不要为了 API 新建一套测试版 pipeline。
不要写西游记、斗罗大陆等特例规则。
不要引入 stopword 式修补。
不要改变 ontology → graph → entity resolution → db → agent profile 的主线。

九、优先实现顺序

第一阶段：
- FastAPI 项目骨架
- POST /projects
- GET /projects/{project_id}/status
- project_store.py
- status.json 写入和读取
- 6MB txt 校验

第二阶段：
- 把现有 pipeline 封装成 run_full_pipeline(project_dir)
- 后台任务调用 pipeline
- status 进度更新
- ready / failed 状态

第三阶段：
- GET /dashboard
- GET /characters
- GET /relationships
- GET /world

第四阶段：
- POST /chat
- chat_runtime.py
- 加载 agent_profile / character_state / world_db
- 调用本地 LLM
- 保存 session history

第五阶段：
- 错误处理
- 日志
